# 埋め込み空間の可視化 設計

- 作成日: 2026-07-20
- 関連 issue: #208（当初は Python スクリプト単体の提案。本設計はそれを PoC と位置づけ、アプリ内ビューまでを射程に入れて再定義する）
- 関連設計: `docs/design/2026-07-17-search-enhancement-design.md`（埋め込み・ベクトル検索の導入）、`docs/design/2026-07-20-classification-observability-design.md`（`classification_log`）

## 1. 背景と目的

`vec_chunks`（sqlite-vec, `float[1024]`）には bge-m3 によるメールチャンクの埋め込みが既に蓄積されている。この高次元ベクトルを 2 次元に落として散布図に描き、**案件ごとにメールがクラスタとして分離しているか**を目視で確認したい。

### 前提: 分類と埋め込みは独立している

設計上きわめて重要な事実として、**Pigeon の AI 分類は埋め込みを一切使っていない**。

- 分類（`classifier/`）は完全に LLM プロンプトベースであり、グルーピングのシグナルは送信者アドレス・最近の件名・案件の階層 path である（`classifier/prompt.rs`）
- 埋め込みが使われているのは検索画面のベクトル検索モードのみ（`SearchModeToggle` → `semantic_search` → `db/vec_search.rs`）

したがって、散布図上で案件の色が混ざっていても、それは分類のバグではない。LLM は埋め込みを見ていないのだから、ズレるのはむしろ自然である。

この可視化が答えるのは次の問いである:

> **bge-m3 の埋め込みが捉える意味空間の構造は、実際の案件区分とどれくらい一致するか。**

一致していれば、埋め込みを将来の分類シグナルとして使う余地がある。一致していなければ、LLM が意味的類似とは別の何か（送信者など）を手がかりにしていることの証拠になる。いずれにせよ、現時点で未検証の仮説に最初の答えを与える。

## 2. スコープ

### 今回のスコープ

埋め込み空間の構造そのものの可視化に限定する。

- 点 = メール（既定）またはチャンク（切替）
- 色 = 割り当てられた案件 / 未分類
- 目的 = クラスタの分離度の目視確認

### スコープ外（将来課題）

以下は同じ散布図の上に重ねられる見込みだが、今回は実装しない。**いずれも「埋め込み空間が信頼できる」ことを暗黙の前提としており、まず空間そのものの検証を済ませる必要がある**ため。

1. **ベクトル検索の探索範囲の可視化**
   クエリの埋め込みを同じ空間に射影し、KNN で拾われた近傍 k 件をハイライトする。「検索条件がどのあたりを探索しているのか」「なぜ意図しないメールが上位に来たのか」を追える。`vec_search.rs` には *KNN の後段で account/project フィルタを適用するため上位 k をスコープ外が占有すると取りこぼす* という既知の制限がコメントで明示されており、この可視化はその実害を目で確かめる手段になる。

2. **LLM 分類の確信度の可視化**
   `classification_log`（v22, #206 で追加）は現在**書き込み専用で読み手が存在しない**。確信度の生の分布を見る手段が実装されていない。2026-07-20 に「確信度が全く校正されておらず 0.9 未満が皆無」という問題を見つけてプロンプトを修正した（#207）が、**その効果測定の手段がまだ無い**。低確信の点が空間上どこに位置するか（クラスタ境界にいるのか、孤立点なのか）を重ねられれば、キャリブレーションの分析軸になる。

3. **ユーザー向け探索 UI への昇格（B フェーズ）**
   本設計は「開発者が挙動を理解するための診断ツール（A）」だが、将来的に「点をクリックしてメールを開き、近傍から関連メールを辿る」ユーザー向け機能へ発展させることを見据える。そのため診断用の情報（確信度、閾値、探索半径）は後から剥がせる形で重ね、コアの描画部分は独立性を保つ。

4. 3D / インタラクティブ（plotly 等）可視化
5. 埋め込み品質の定量評価（クラスタ純度スコア等）

## 3. 全体構成: PoC を先行させる二段構え

**PoC（Python）を完成させてから、アプリ内ビューに着手する。** 並行しない。

PoC の役割は単なる先行実装ではなく、**PCA の較正**である。アプリ側は PCA しか持たない設計のため、PoC で PCA と t-SNE を並べて描き、「PCA でどこまで見えるか」を確認する必要がある。

- PCA でも案件の塊が見える → アプリ側の PCA 実装で目的を達成できる
- PCA では全く見えず t-SNE でのみ見える → アプリ側の設計を見直す（後述の代替案 C を再検討）

この分岐判断のために PoC を先に終わらせる。

```
[PoC] scripts/visualize_embeddings.py
        DB直読み → numpy → PCA / t-SNE / UMAP → PNG
                              ↓ 較正結果を受けて
[アプリ]
  Rust: embedding_map モジュール
        ├─ ベクトル取得（vec_chunks JOIN）
        ├─ メール単位集約（chunk centroid）/ チャンク単位 の切替
        └─ べき乗法で上位2主成分のみ算出 → 2D座標
                              ↓ Tauri command（座標のみ, 1点 ~50B）
  React: 独立 WebviewWindow
        └─ Canvas 自前描画（可視化ライブラリ非導入）
```

## 4. 設計判断とその理由

### 4.1 次元削減はどこでやるか → アプリ側は Rust で PCA を自前実装

検討した案:

| 案 | 内容 | 判断 |
|---|---|---|
| **A（採用）** | PoC で検証し、アプリは PCA を Rust 実装 | 依存ゼロ・高速・自己完結。座標が決定的で再現性がある |
| B | UMAP 相当まで Rust 実装 | 実装コストが数日規模。品質検証という目的に対して過剰 |
| C | Python がバッチで座標を計算し DB に保存、アプリは描画のみ | 可視化品質は最良だが Python 依存が残り、ユーザー向け（B フェーズ）に進めない |

t-SNE / UMAP の実用的な実装は Python エコシステムにしか無く、Rust にも JS にも移植する現実的な選択肢が無い。一方、目的である「案件が分離しているか」の判定には PCA でも十分に答えが出る見込みが高い。**分離が良ければ PCA でも塊が見えるし、PCA で全く見えないなら t-SNE で無理に分けても実用性は疑わしい。**

C は Python 依存が残るため B フェーズへの道を塞ぐ。PoC の較正結果が芳しくない場合の退避先として記録に留める。

### 4.2 PCA の算出方法 → べき乗法（power iteration）

共分散行列（1024×1024）を構築すると計算量が `O(N × 1024²)` となり、N = 1万で約100億回の積和 → Rust でも数秒かかる。

**上位 2 主成分しか必要ないため、共分散行列を作らずべき乗法で直接求める。** 計算量は `O(N × 1024 × 反復数)` に下がり、N = 1万・20 反復で 0.2 秒程度に収まる。第 2 主成分は第 1 主成分の方向を除去（デフレーション）してから同じ手順で求める。

### 4.3 表示粒度 → メール単位を既定、チャンク単位に切替可能

| 粒度 | 内容 | 位置づけ |
|---|---|---|
| **メール単位（既定）** | メールの全チャンクの平均ベクトル（centroid）を 1 点とする | 点数が減って軽く、1 メール = 1 点なので案件ごとの塊が読みやすい。**目的に直結する** |
| チャンク単位（切替） | チャンクごとに 1 点。1 メールが複数点に分かれる | 埋め込みの生の姿。「なぜこのメールが変な位置にいるのか」を掘るときに効く |

Rust 側は同じクエリで集約の有無を切り替えるだけなので、切替の追加コストは小さい。centroid の平均化で情報が落ちる（話題が 2 つ混ざったメールが中間点に来る）という短所に対する安価な保険となる。

### 4.4 件数上限 → 既定 1000 件、容易に変更可能とする

1000 という数字自体に根拠はない。**重要なのは、いつでも簡単に変えられること。** そのため二重に変更手段を用意する:

- Rust 側に既定値の定数を 1 つ置く（`vec_search.rs` の `KNN_FACTOR` / `KNN_MAX` と同じ流儀）
- Tauri command の引数で上書き可能にし、ウィンドウ上にも件数入力を出す → **再ビルドなしで変更できる**

なお 4.2 のべき乗法を採用すれば全件処理でも 1 秒以内に収まる見込みであり、上限は性能上の必須要件ではない。まず 1000 件で始め、実データを見ながら調整する。

**PoC 実装（Python CLI）での適用範囲**: 上限は「未分類メールも含める」場合にのみ適用する。案件割り当て済みのみ（既定）のときは全件読む。`mail_id` が UUID 相当の TEXT PK であるため、件数で機械的に切ると辞書順の偶然で小さな案件が丸ごと欠落しうる。Task 7（小規模案件が分離して見えるか）はまさにこの取りこぼしに弱いため、割り当て済み側では上限を掛けない。未分類側は 1 割程度の少数派プロジェクトを埋もれさせないための除外がすでにあり、含めた場合の点数爆発を抑える目的でのみ上限を使う。

### 4.5 置き場 → Tauri の独立 WebviewWindow

現状のフロントはルーターを持たず、ビュー切替は `uiStore.viewMode` の単一 state を `App.tsx` の三項演算子ネストで分岐している。中央ペインは `w-80`（320px 固定）で、散布図には狭すぎる。

独立ウィンドウを選ぶことで:

- サイズが自由になり、メイン画面と並べて見られる
- 既存の 3 ペインレイアウトと `viewMode` の分岐に一切手を入れない
- **zustand のストアが共有されないことが、ここでは長所に転じる。** 可視化ウィンドウは自分専用の状態を持ち、必要なデータは Tauri command で直接引く。メインウィンドウとの結合がゼロになり、B フェーズへ発展させる際も独立性が保たれる

### 4.6 描画 → Canvas 自前実装（可視化ライブラリを導入しない）

`package.json` には d3 / recharts / plotly / chart.js のいずれも存在しない。Pigeon は依存を絞る方針であり、散布図・ズーム・パン・ホバー程度であれば自前実装で足りる。数千点規模では SVG より Canvas が向く。**軽さを優先する。**

## 5. データフロー

1. ウィンドウ起動 → command 呼び出し（上限 = 既定値、粒度 = メール単位）
2. Rust: `vec_chunks` × `mail_chunks` × `mails` × `mail_project_assignments` × `projects` を JOIN してベクトルとラベルを取得
   - 埋め込みは `zerocopy::IntoBytes` で生バイト保存されているため、`&[f32]` として復元する（`db/chunks.rs` の保存に対応）
3. メール単位の場合は chunk centroid を算出
4. 全体を中心化 → べき乗法で PC1 / PC2 を求め、各点を 2 次元に射影
5. `{x, y, mail_id, subject, project_id, project_name}` の配列を返す
6. Canvas に描画。案件ごとに色、未分類はグレー

**1024 次元のベクトルはフロントに渡さない。** 1 万点で 40MB に達し IPC が詰まる。Rust 側で 2 次元まで落とし、座標のみを送る（1 点あたり約 50 バイト、1 万点でも数 MB 程度）。

## 6. インタラクション

- **ズーム / パン**: 密集部を拡大して確認する
- **ホバー / クリック**: **件名のみ**を表示する。本文は出さない（診断目的には件名で足り、本文表示は B フェーズの検討事項）
- **案件ハイライト**: 特定の案件を強調し、それ以外をグレーに落とす。点を減らすのではなく色を落とすだけなので再計算が不要で、コストがかからない

## 7. エラー処理

| 状況 | 挙動 |
|---|---|
| 埋め込みが 0 件 | 「埋め込みが未生成」を明示する。Ollama 未起動時に埋め込みパスが静かに打ち切られる既存挙動（`AppError::OllamaConnection` で `break`）があるため、実際に起こりうる |
| 点が 1〜2 件 | PCA が定義できないため、その旨を表示して描画しない |
| べき乗法が収束しない | 反復上限で打ち切り、その時点の近似を使う（実用上問題なし） |
| sqlite-vec 拡張の未ロード | PoC 側の主な失敗要因。`vec0` 仮想テーブルの読み取りに必須である旨を README に明記する |

## 8. テスト方針

TDD に従い、テストを先に書く。

**Rust**
- べき乗法 PCA を**既知の答えを持つ合成データ**で検証する（明確に 2 軸方向へ伸びた点群を入力し、主成分が期待どおりの向きを向くか）
- chunk centroid 集約の単体テスト
- 空・1 件・2 件のエッジケース

**フロントエンド**
- 座標 → スクリーン変換、ズーム / パンの変換行列
- ヒットテスト（点のクリック判定）
- Canvas への実描画は目視確認とする

**PoC**
- テスト対象外。使い捨ての調査ツールと位置づける

## 9. 成果物

**第 1 段階: PoC**
- `scripts/visualize_embeddings.py` — DB 直読み、PCA / t-SNE / UMAP の比較、PNG 出力
- `scripts/requirements.txt` — numpy, scikit-learn, matplotlib, sqlite-vec（任意で umap-learn）
- `scripts/README.md` — DB パスの指定方法、実行手順、手法の使い分け（PCA = 高速・大局、t-SNE = クラスタ分離、UMAP = バランス）

DB パスは `{dirs::data_dir()}/Pigeon/pigeon.db`（`src-tauri/src/lib.rs`）。`--db` 引数で上書き可能にする。

**第 2 段階: アプリ内ビュー**（PoC の較正結果を確認してから着手）
- Rust: 座標算出モジュール（ベクトル取得・集約・べき乗法 PCA）
- Rust: Tauri command（上限・粒度を引数に取り、座標とラベルを返す）
- Front: 独立 WebviewWindow と Canvas 描画コンポーネント

## 10. 確認方法

- **PoC**: 実データでスクリプトを実行し PNG が生成されること。案件が複数ある DB で、PCA と t-SNE を並べて分離の見え方を比較すること
- **アプリ**: ウィンドウを開いて散布図が描画され、案件ごとの色分け・ズーム・件名表示・粒度切替が動作すること
