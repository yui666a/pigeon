# 検索強化（正規化・ベクトル検索・スマートビュー）設計書

作成日: 2026-07-17
ステータス: レビュー待ち

## 背景と課題

ペルソナ「アスカ」（[docs/personas.md](../personas.md) 参照）は、1つの舞台公演に関する演出・ケータリング・音楽・照明など全領域のメールを受信し、その中から自分の担当（照明）に関するメールだけを瞬時に見つける必要がある。

現状の検索は SQLite FTS5（trigram トークナイザ）＋ 3文字未満クエリの LIKE フォールバックによる**部分文字列一致のみ**であり（`src-tauri/src/db/search.rs`）、以下の揺れに対応できない。

| 揺れの種類 | 例 | 現状 |
|---|---|---|
| 全角/半角・大文字/小文字 | `SATO` / `sato` / `ＳＡＴＯ` | ヒットしない（正規化なし） |
| ひらがな/カタカナ | `さとー` / `サトー` | ヒットしない |
| ローマ字/カナ/漢字 | `SATO` / `サトー` / `佐藤` | ヒットしない |
| 同義語 | `プリンター` / `端末` / `デバイス` | ヒットしない |

さらに「現場マネージャーが全領域を1通に取りまとめて送る」運用があるため、送信者ベースの領域判別は成立せず、**本文内容ベース**のアプローチが必須である。

## ゴール

1. **正規化**: 全角/半角・かな種・大小文字の揺れを文字列検索で吸収する
2. **ベクトル検索**: 同義語・読みが同じ揺れを意味ベースの検索で吸収する（ローカル埋め込みモデル）
3. **スマートビュー**: 「照明」のような担当領域の検索を保存し、サイドバーからワンクリックで開けるようにする

## 非ゴール（将来課題）

- **RAG（質問応答・要約）**: 索引はRAG対応可能な形にするが、機能自体は別設計とする
- **ハイブリッド検索（FTS＋ベクトルのスコア融合）**: v1 はモード切替で分離。融合は第3のモードとして将来追加可能
- **連絡先エイリアス**: `SATO`↔`佐藤` のようなローマ字↔漢字の同定は埋め込みでも完全には解けない。連絡先に別名を持たせるクエリ展開で補完する（別設計）
- **スマートビューの未読バッジ・新着通知・案件スコープ絞り込み**

## 全体像

3フェーズに分け、各フェーズを独立した PR スタックで実装する。

```
Phase 1: 正規化層          … 文字列検索の改善（既存FTS5の索引再構築）
Phase 2: ベクトル検索       … sqlite-vec + Ollama 埋め込み（スパイク検証後）
Phase 3: モード切替UI + スマートビュー
```

## Phase 1: 正規化層

### 正規化仕様

`normalize_for_search(text: &str) -> String` を `src-tauri/src/search_normalize.rs`（または db 配下）に新設する。処理は順に:

1. Unicode NFKC 正規化（全角英数→半角、半角カナ→全角カナ、㈱→(株) 等）
2. ひらがな→カタカナ統一（`ぁ..ゖ` → `ァ..ヶ` のコードポイントシフト）
3. ASCII 小文字化（`to_lowercase`）

この関数を**索引時**（`fts_mails` への挿入内容）と**クエリ時**（検索語）の両方に適用する。LIKE フォールバック経路にも同じ関数を適用する。

### 索引の変更

- migration（次の空き番号、執筆時点で v17）で `fts_mails` を正規化済みテキストで再構築する
- 既存のトリガー（mails INSERT/DELETE 同期）は SQL トリガーのままでは Rust の正規化関数を呼べないため、**トリガーを廃止し、メール保存/削除処理の Rust 側で fts_mails を明示的に更新する方式に変更**する。これにより索引内容を常に正規化済みに保てる

### スニペット表示

`fts_mails` に正規化済みテキストを格納するため、FTS5 の `snippet()` は正規化後の文字列（カタカナ・小文字化済み）を返してしまう。表示品質を保つため、スニペットは自前生成に切り替える:

- 正規化時に「正規化後オフセット→原文オフセット」の対応表を保持できるよう、正規化は1文字単位のマッピングとして実装する（NFKC で長さが変わる文字はマッピングを保持）
- 検索ヒット時、正規化済み本文中のクエリ位置を求め、対応表で原文位置に変換し、原文からスニペットを切り出してハイライト範囲を返す

### テスト（TDD）

- 正規化関数の単体テスト: 全半角・かな種・大小文字・混在ケース、オフセット対応表の正確性
- 検索統合テスト: `ＳＡＴＯ` で `sato` がヒット、`さとー` で `サトー` がヒット、スニペットが原文で返る

## Phase 2: ベクトル検索

### 構成

```
メール同期 ─→ チャンク化（引用除去・件名プレフィックス）─→ Ollama /api/embed ─→ sqlite-vec 索引
検索クエリ ─→ 正規化なし（原文のまま）─→ Ollama /api/embed ─→ KNN検索 ─→ mail_id でグルーピング ─→ メール一覧
```

### 埋め込みモデル

| 項目 | 内容 |
|---|---|
| デフォルト（スパイクで確定） | **bge-m3**（Ollama 公式ライブラリ、`ollama pull bge-m3` 一発、1024次元、最大8192トークン、プレフィックス不要、MIT） |
| 将来差し替え候補 | **Ruri v3-130m**（cl-nagoya、JMTEB Retrieval 81.89、Apache-2.0、512次元）— 忠実な GGUF が入手可能になった時点で再評価 |
| 前提 | Ollama v0.31.2 以降 |

モデルは**設定で差し替え可能**にする。設定項目: モデル名・ベクトル次元・クエリ/文書プレフィックス（Ruri v3 は `検索クエリ: ` / `検索文書: ` の付与が必須、bge-m3 は不要）。**モデル変更時は全チャンクの再埋め込みが必要**（次元も変わるため索引テーブルを作り直す）。

### スパイク（Phase 2 実装前の検証）

Ruri v3-130m を採用してよいか、実装前に以下を確認する:

1. Ollama v0.31.2+ で Ruri v3 GGUF が動作するか
2. sentence-transformers の出力と cos 類似度がほぼ 1.0 で一致するか（GGUF 変換の品質検証）
3. 実データで「サトー↔佐藤」「プリンター↔端末」が上位ヒットするか

いずれかが不合格なら bge-m3 をデフォルトにして先行リリースし、Ruri v3 の安定後に差し替える。

**スパイク結果（2026-07-17 実施 → 不合格、bge-m3 をデフォルトに確定）:**

- 環境: Ollama v0.32.1（0.31.1 から更新して検証）
- 条件1: **合格** — コミュニティ GGUF（keisuke-miyako/ruri-v3-130m-gguf-q8_0、136MB）の import 成功、`/api/embed` が 512 次元を正常返却
- 条件2: **不合格** — 本家 sentence-transformers との cos 類似度が実測 0.74〜0.89（5件全て閾値 0.99 未達）
- 条件3: 未検証（条件2不合格によりスキップ）
- 原因: モデル重みの量子化誤差ではなく **GGUF 変換のトークナイザ忠実性**。同一文字列が本家トークナイザで12トークン、GGUF 経由で17トークン（GGUF メタデータが `tokenizer.ggml.pre: default` の汎用 pre-tokenizer になっており、Ruri v3 のカスタム BPE 語彙が再現されていない）。pooling 方式（MEAN）は両者一致で原因ではないことを確認済み
- 判定: Ruri v3 自体の能力ではなく変換品質の問題のため、「安定後差し替え」の将来パスは維持。忠実な GGUF（公式配布 or 検証済み自前変換）が得られた時点で再評価する

### チャンク化

- **引用除去**: `>` プレフィックス行、`On ... wrote:` / `----- Original Message -----` 等の引用ブロックを除去してから分割する
- **分割**: 文字数ベースで約800文字（≒400〜500トークン相当）、段落境界優先、オーバーラップ約100文字
- **件名プレフィックス**: 各チャンクの先頭に `件名: {subject}\n` を付与してから埋め込む
- 短いメールは 1メール = 1チャンクになる（大半のケース）

### DB スキーマ

sqlite-vec 拡張を rusqlite にリンクして使用する。

```sql
-- チャンク本体（RAG-ready: メタデータと原文チャンクを保持）
CREATE TABLE mail_chunks (
  id INTEGER PRIMARY KEY,
  mail_id INTEGER NOT NULL REFERENCES mails(id) ON DELETE CASCADE,
  chunk_index INTEGER NOT NULL,
  content TEXT NOT NULL,           -- 件名プレフィックス込みの埋め込み対象テキスト
  embedded_at TEXT,                -- NULL = 未埋め込み（キュー対象）
  UNIQUE(mail_id, chunk_index)
);

-- ベクトル索引（sqlite-vec 仮想テーブル。次元はモデル設定に依存）
CREATE VIRTUAL TABLE vec_chunks USING vec0(
  chunk_id INTEGER PRIMARY KEY,    -- mail_chunks.id
  embedding float[1024]            -- 次元はモデル設定に依存（bge-m3 = 1024）
);
```

`project_id` / `account_id` は `mails` への JOIN で解決する（非正規化しない）。

### 増分埋め込みとバックフィル

- メール同期完了後、`embedded_at IS NULL` のチャンクをバックグラウンドタスクが順次埋め込む（既存の同期パイプラインの後段に追加）
- 初回有効化時は全既存メールのバックフィルが走る。進捗（n/全件）を設定画面に表示し、アプリ操作はブロックしない
- Ollama 未起動・モデル未導入の間はキューに滞留するだけで、エラーにはしない

### 検索 API の層分け（RAG-ready）

- **内部層** `search_chunks(query) -> Vec<ChunkHit>`: クエリ埋め込み→KNN（top-K=50 程度）→チャンクとスコアを返す。将来の RAG はこの層を入口にする
- **UI層** `search_mails_semantic(account_id, query) -> Vec<MailHit>`: `search_chunks` の結果を mail_id でグルーピングし、メールごとの最高チャンクスコアを採用して関連度順に返す

### UX（ベクトルモード選択時）

- モデル未導入の場合: 案内ダイアログを表示し、モデルのダウンロード（`ollama pull` 相当、Ruri v3 の場合はインポート手順）へ誘導する
- Ollama 未起動の場合: 既存の LLM 分類と同じエラーハンドリングに従う

### テスト（TDD）

- 引用除去・チャンク分割の単体テスト（日本語メール実例パターン）
- 埋め込みプロバイダを trait 化し、テストでは決定的なフェイク埋め込みで KNN→グルーピングの統合テスト
- モデル設定変更時の再索引フローのテスト

## Phase 3: 検索モード切替 UI + スマートビュー

### 検索モード切替

- `SearchBar` の近傍にモードトグル（`文字列` / `ベクトル`）を追加する。デフォルトは `文字列`
- `searchStore` に `mode: 'fulltext' | 'semantic'` を追加し、モードに応じて `search_mails` / `search_mails_semantic` を呼び分ける
- 選択したモードは永続化する（次回起動時も維持）

### スマートビュー（保存検索）

**コンセプト: 「検索の保存」であり、新しい概念を増やさない。**

- 検索結果画面に「この検索を保存」ボタンを置き、名前を付けて保存する（例: `照明`）。クエリとモードをセットで保存する
- サイドバーの案件ツリーとは**別セクション**「スマートビュー」に一覧表示する（案件=公演軸、スマートビュー=担当領域軸の直交する2軸）
- クリックで保存されたクエリを再実行し、メール一覧を表示する。並び順は関連度順/日付順を切替可能
- 検索ベースなので誤分類ラベルが固定される問題は起きない（精度不足は順位の乱れとして現れるだけ）
- 削除・リネームはコンテキストメニューから行う

```sql
CREATE TABLE saved_searches (
  id INTEGER PRIMARY KEY,
  name TEXT NOT NULL,
  query TEXT NOT NULL,
  mode TEXT NOT NULL CHECK (mode IN ('fulltext', 'semantic')),
  sort_order INTEGER NOT NULL DEFAULT 0,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```

### テスト（TDD）

- モード切替と呼び分けのコンポーネント/ストアテスト（Vitest）
- saved_searches CRUD の Rust テスト、スマートビュー表示のコンポーネントテスト

## セキュリティ

- 埋め込み生成・ベクトル検索は**すべてローカル**（Ollama）で完結する。埋め込みは本文全文をチャンク化して処理するため、クラウド埋め込みAPIは「LLM送信は本文冒頭1000文字まで」のポリシーと矛盾する。**クラウド埋め込みは提供しない**
- ベクトルと原文チャンクは SQLite（ローカル）に保存する。機密情報の扱いは既存の mails テーブルと同等

## 実装順序と PR 分割

1. スパイク（ブランチのみ、マージ不要）: Ruri v3 の Ollama 動作検証
2. PR: Phase 1 正規化層（正規化関数 → FTS 再構築 migration → スニペット自前生成）
3. PR: Phase 2 基盤（sqlite-vec 導入・チャンク化・埋め込みキュー）
4. PR: Phase 2 検索（semantic search command・モデル設定）
5. PR: Phase 3 モード切替 UI
6. PR: Phase 3 スマートビュー

依存関係が一直線のため、必要に応じて Stacked PR とする。マイグレーション番号は各 PR 作成時点の空き番号を採番する（並行開発時の番号衝突に注意）。
