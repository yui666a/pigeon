# 埋め込みマップ（アプリ内ビュー）設計

- 作成日: 2026-07-22
- 位置づけ: `docs/design/2026-07-20-embedding-visualization-design.md` の**第2段階**。同設計書の PoC（第1段階, PR #226/#227）で「PCA で案件が分離する」ことを較正済み（§11）。本書はその結果を受け、アプリ内のエンドユーザー向けビューを設計する
- 関連: `docs/design/2026-07-17-search-enhancement-design.md`（埋め込み・vec_chunks の導入）、`docs/adr/0004-ai-native-dispatch-architecture.md`（UseCase/dispatch 境界）

## 1. 目的

メール埋め込み（bge-m3, `vec_chunks`, float[1024]）を 2 次元に落として案件ごとに色分けした散布図を、**アプリ内の独立ウィンドウ**で見られるようにする。エンドユーザーが対象。

第1段階（PoC）は「開発者が bge-m3 の分離度を検証する」ためのものだったが、本段階は**メールを使う人が使う道具**であり、目的が異なる。

### 用途（優先順）

1. **未分類の発見（第一目的）**: 未分類メールが既存案件の塊の近くにいることに気づき、その場で案件に片付ける。**システムは割り当てを提案せず、「見てみて」のきっかけを与えるだけ。判断はユーザー**（PoC の較正は「案件全体の分離」を確認したもので、「個々の未分類が正しい案件に近い」ことは未検証。未検証の仮定に乗らない）。
2. **分類の健康診断（第二目的）**: 案件ごとに点が固まっていれば「AI がちゃんと分けている」と納得でき、混ざっている点で「ここは怪しいかも」と気づける。

### 非目的（将来）

- メールを空間的に探すナビゲーション（用途としての「地図」）は主眼にしない
- 埋め込みの近さに基づく自動割り当て提案（较正で未検証のため踏み込まない）
- 検索の探索範囲・確信度分布の可視化（PoC 設計書 §2 の将来課題のまま）

## 2. 全体像

```
[別ウィンドウ] Tauri WebviewWindow "embedding-map"（visualization.html）
  ├─ 左〜中央: Canvas 散布図
  │    点 = メール（centroid）。案件 = 色、未分類 = 背面の薄いグレー
  │    ズーム / パン / 点ホバー・クリック
  ├─ 右: 案件パネル（D&D ドロップ先） + 本文プレビュー（軽量）
  └─ 状態: このウィンドウ専用の zustand（メインと非共有）

[Rust] 新規 Tauri command 2 本
  ├─ embedding_map_points
  │    vec_chunks から生ベクトル読み(bytes→f32) → メール単位 centroid 集約
  │    → べき乗法 PCA で 2D → 点データ配列を返す
  └─ mail_preview(mail_id) → 件名/送信者/日付/本文冒頭 を返す軽量 command

[割り当て] 既存 bulk_move_mails を別ウィンドウから直 invoke
  → 成功 → emit("mail-assigned", {mail_id, project_id})
  → メインウィンドウが listen → 未分類/案件を再取得
```

## 3. 主要な流れ（発見 → 片付け）

1. ユーザーがウィンドウを開く → 全メールの散布図が描画される（案件 = 色、未分類 = 背面グレー）
2. 未分類の点が案件の塊の近くにいるのに気づく
3. その点をクリック → 右に軽量プレビュー（件名・送信者・日付・本文冒頭）
4. 判断して、点を右の案件パネルの案件へドラッグ → `bulk_move_mails([mail_id], project_id)`
5. メインウィンドウの未分類リスト・案件一覧が自動で更新される

## 4. 設計判断とその理由

各判断は 2026-07-22 のブレストで、既存資産の調査（`src-tauri/`, `src/` の実地確認）を踏まえて確定した。

### 4.1 置き場 → Tauri 独立ウィンドウ

現状マルチウィンドウは一切使っておらず（`tauri.conf.json` の window は 1 定義、capability も `["main"]`、`WebviewWindowBuilder` の使用ゼロ）、独立ウィンドウは新規構築になる。それでも独立ウィンドウを選ぶ理由は、**メイン画面と並べて、片方でマップを見ながら作業したい**というユーザー体験を優先するため。

トレードオフとして、独立ウィンドウは zustand ストアを共有しない。これは §4.4 の割り当て設計に影響する（ストア経由でなく command を直 invoke する）。

### 4.2 座標計算 → Rust でべき乗法 PCA（依存追加なし）

PoC の較正で PCA が案件を分離できると確認済みのため、t-SNE / UMAP は不要。上位 2 主成分だけ必要なので、共分散行列（1024×1024）を構築せず**べき乗法（power iteration）で直接求める**。計算量は `O(N × 1024 × 反復)` で、N が数千でも 1 秒未満。第 2 主成分は第 1 主成分をデフレーションしてから同手順で求める。

- 線形代数クレート（nalgebra 等）は導入しない。上位 2 成分のためにクレートを足すのは過剰
- `nalgebra` 案・フロント計算案は却下。前者は依存過剰、後者は 1024 次元×数千点を IPC で運ぶことになりシリアライズで固まる（PoC 設計書 §5 の原則「1024 次元をフロントに渡さない」に反する）

**生ベクトルの読み出しは新規**。既存コードは `vec_chunks` から距離しか読んでおらず、生の 1024 次元ベクトルを bytes→f32 で復元する関数は前例がない。書き込みは `zerocopy::IntoBytes`（`chunks.rs`）なので、`vec0` が返す raw bytes を `f32::from_le_bytes`（`chunks_exact(4)`）または `zerocopy` で復元する。

### 4.3 表示するデータ → 全メール（案件 + 未分類すべて）

未分類は実データで 9 割超（PoC 時点で 863 割り当て / 約 1.15 万通）。まず全体像をそのまま見て、実際の見え方を確かめる。未分類が案件を覆い隠す問題は**描画のレイヤリングで緩和**する（§4.6）。

点数は数千〜1 万規模になるが、べき乗法 PCA も Canvas 描画も 1 万点を問題なく扱える。実際に見て「絞りたい」となれば、ウィンドウ内の件数フィルタUIを後から足す（今回は入れない — YAGNI）。

点は**メール単位**（そのメールの全チャンクの centroid）。理由は PoC 設計書 §4.3 と同じ（案件の塊が読みやすい）。バックエンドの割り当ても全て mail_id 単位なので整合する。

### 4.4 割り当て → 既存 command を直 invoke + イベント同期

調査の結果、**割り当ての全層（command / store / DB 書き込み）が mail_id 単位で既存**であり、`bulk_move_mails(mailIds, projectId)` に単一 ID を渡すだけで 1 通割り当てられる（`correction_log` 記録・スレッド追従も内包）。**新規の割り当て command は不要**。

ただし §4.1 の通り別ウィンドウは zustand を共有しないため、`useMailStore.bulkMoveMails` をそのまま呼べない。**このウィンドウは `bulk_move_mails` command を直接 invoke する**（store を経由しない）。

割り当て後の同期は Tauri イベントで行う:
- 別ウィンドウ: 割り当て成功 → `emit("mail-assigned", { mail_id, project_id })`
- メインウィンドウ: `listen("mail-assigned")` → 未分類リスト・案件一覧を再取得

既存のイベント基盤（`embed-progress` 等で実績あり）に乗る。

「同期しない」案は却下。「別ウィンドウで割り当てたのにメインの未分類にまだ居る」という混乱を避ける。

### 4.5 割り当て UI → ウィンドウ内に案件パネル（D&D ドロップ先）

別ウィンドウ内で発見 → 本文 → 割り当てを完結させる。メインの案件ツリー（左サイドバー）は別ウィンドウから到達できないため、**このウィンドウ内に案件パネルを持ち、そこをドロップ先にする**。

既存の D&D 基盤はマウスイベントベースの自前実装（`useMailDrag` / `dragStore` / `DragOverlay` / droppable な `ProjectListItem`）で、mail_id 配列を運ぶだけ。ただし別ウィンドウでは zustand（`dragStore`）が共有されないため、**D&D の考え方は踏襲しつつ、このウィンドウ専用に再実装**する（`useMailDrag` のロジックは移植可能）。

### 4.6 描画 → 素の Canvas、未分類を背面レイヤーに

可視化ライブラリは導入しない（現状 d3/canvas 系ゼロ、Pigeon は依存を絞る方針）。素の `<canvas>` + `getContext("2d")` で散布図・ズーム・パン・ホバーを書く。

未分類が案件を覆い隠さないよう、PoC の `plot.py` と同じレイヤリングを持ち込む: **未分類は背面・薄いグレー（`#cccccc` 相当）・小さめ、案件は前面・色付き・大きめ**。

案件色は既存の規則 `project.color ?? "#6b7280"` を踏襲する（PoC の tab20 自動配色ではなく、ユーザーが設定した案件色に揃える）。自動パレットは存在しない。

### 4.7 本文プレビュー → 軽量（件名・送信者・日付・本文冒頭）

既存の `MailView`（`MailBody` + cid 画像 + 添付 + DOMPurify）は作り込まれており別ウィンドウへの移植は重い。発見の判断は件名 + 送信者 + 本文冒頭でほぼ付くため、**軽量プレビュー**に留める。

- 新規の軽量 command `mail_preview(mail_id)` が 件名/送信者/日付/本文冒頭（数百文字）を返す
- フル本文が必要なときは「メインで開く」導線を 1 つ置く（点の mail_id をメインに渡し、メイン側で既存の `selectMail` を発火）

## 5. コンポーネントと責務

| 層 | 追加物 | 責務 |
|---|---|---|
| Rust | 生ベクトル読み出し関数（`db` 層） | `vec_chunks` から 1024 次元 f32 を復元。案件ラベルと JOIN |
| Rust | べき乗法 PCA（新モジュール） | centroid 集約 → 上位 2 主成分 → 2D 射影 |
| Rust | command `embedding_map_points` | 上記を束ね、点データ配列を返す |
| Rust | command `mail_preview` | mail_id → 件名/送信者/日付/本文冒頭 |
| フロント | 別 HTML エントリ（`visualization.html`）+ Vite 第2入力 | 独立ウィンドウの中身 |
| フロント | Canvas 散布図コンポーネント | 描画・ズーム・パン・ホバー・クリック |
| フロント | 案件パネル（ドロップ先） | 案件一覧 + D&D 受け |
| フロント | 本文プレビュー | 軽量プレビュー表示 |
| フロント | ウィンドウ専用 store | 点データ・選択状態（メイン非共有） |
| 配線 | `tauri.conf.json` + `capabilities` | 新ウィンドウラベル `embedding-map` |
| 配線 | ウィンドウを開く導線 | メイン側にボタン（未分類ビュー付近） + `WebviewWindow` 生成 |

## 6. データフロー

1. メイン側の導線からウィンドウを開く（`WebviewWindow` で `visualization.html` をロード）
2. ウィンドウ起動 → `embedding_map_points` を invoke
3. Rust: `vec_chunks` × `mail_chunks` × `mails` × `mail_project_assignments` × `projects` を JOIN し、生ベクトル + ラベルを取得
4. メール単位 centroid → 中心化 → べき乗法で PC1/PC2 → 各点を 2D 射影
5. `{ x, y, mail_id, subject, project_id, project_name }[]` を返す（1024 次元は返さない）
6. Canvas 描画。案件色（`project.color ?? "#6b7280"`）、未分類は背面グレー
7. 点クリック → `mail_preview(mail_id)` → 右にプレビュー
8. 点を案件パネルへドラッグ → `bulk_move_mails([mail_id], project_id)` → `emit("mail-assigned")`
9. メインが `listen` → 未分類/案件を再取得

## 7. エラー処理

| 状況 | 挙動 |
|---|---|
| 埋め込みが 0 件 | 「埋め込みが未生成」を表示。Ollama 未起動時に埋め込みパスが静かに打ち切られる既存挙動があるため実際に起こりうる |
| 点が 1 件以下 | PCA が定義できないため、その旨を表示して描画しない |
| べき乗法が収束しない | 反復上限で打ち切り、その時点の近似を使う |
| 割り当て失敗（command エラー） | トースト等でエラー表示。楽観的更新は行わず、成功時のみ点の見た目を更新 |
| ウィンドウ多重起動 | 既存ウィンドウがあればフォーカスするだけ（`get_webview_window("embedding-map")` で確認） |

## 8. テスト方針

TDD に従う。

**Rust**
- べき乗法 PCA を**既知の答えを持つ合成データ**で検証（2 軸に伸びた点群 → 主成分の向き）
- 生ベクトル読み出し（bytes→f32 復元）の単体テスト。次元不一致行のスキップ
- centroid 集約の単体テスト
- 空・1 件のエッジケース

**フロント**
- 座標 → スクリーン変換、ズーム/パンの変換
- ヒットテスト（点クリック判定）
- Canvas 実描画は目視確認
- 割り当て → emit の発火、メイン側 listen の再取得（イベント配線の単体テスト）

## 9. 段階の切り出し

本設計は「マップ表示 + 発見 + ウィンドウ内割り当て」を 1 つのまとまりとする。実装計画では、独立ウィンドウという新規基盤のリスクを踏まえ、**先にウィンドウ + 描画 + 点クリックプレビューを立ち上げ、その上に D&D 割り当てとイベント同期を載せる**順で分割する（詳細は実装計画で）。

## 10. 確認方法

- 実データでウィンドウを開き、散布図が描画され案件ごとに色分けされること
- 未分類の点が案件の近くにいるケースを見つけ、クリック → プレビュー → 案件へドラッグ → 割り当てできること
- 割り当て後、メインウィンドウの未分類リストが自動更新されること
- ズーム・パン・ホバーが動作すること
