# Pigeon の開発スタイル

この文書は、2026-07-12 の大規模実装（PR #70〜#86: 送信、既読、添付、削除とアーカイブ、IDLE、通知、ドキュメント整備）で実際に運用した進め方を記録する。
コーディング規約そのものは `agent.md` に、残タスクは `docs/BACKLOG.md` にある。
この文書が扱うのは、それらをどう回すかという作業の型である。

## 全体の流れ

1機能は次のサイクルで進める。

1. 設計書を書く（`docs/superpowers/specs/YYYY-MM-DD-<機能名>-design.md`）
2. テストを先に書き、実装する（TDD）
3. 意味のある粒度でコミットし、PRを作る
4. スタック全体でテストを通してからマージする
5. 実装しなかったことを設計書とバックログに残す

複数機能を同時に進めるときは、このサイクルを機能ごとにサブエージェントへ割り当て、統合だけをリード（メインセッション）が行う。

## 設計書ファースト

コードより先に設計書を書く。
設計書には、採用した方式だけでなく**スコープ外**と**v1の制限**を明記する。
制限を書き残す理由は、後続の作業者（次セッションのエージェントを含む）が「未実装」と「意図的な見送り」を区別できないと、同じ調査を繰り返すからである。

実例として、送信機能の設計書はリッチテキストを明示的にスコープ外とし、削除とアーカイブの設計書は「Sentフォルダはuidが推定値のためサーバー反映しない」という制限を理由付きで残した。
この記録が後日、バックログの最優先項目（APPENDUID/COPYUID対応）の根拠になった。

## テスト駆動開発

Red、Green、Refactor の順で書く。
このプロジェクトで効いた具体的な型は次の二つである。

- **純関数への切り出し**：IMAPやSMTPのような外部通信は自動テストできない。そこで判断ロジック（メッセージ構築、References導出、バックオフ計算、フォルダ判定）を通信から切り離した純関数にし、そこへテストを集中させる。通信そのものは統合境界としてテスト対象外と割り切り、その旨をPRに書く
- **接続の注入**：IDLE監視ループのような状態遷移は、接続部分をクロージャで注入可能にして、切断や再接続のシナリオを台本として流し込んでテストする

検証は次の4点をすべて通してから完了と見なす。

```bash
cd src-tauri && cargo test --lib   # Rust
pnpm test                          # フロントエンド (Vitest)
pnpm exec tsc --noEmit             # 型チェック
pnpm build                         # ビルド
```

テストが通ったと主張する前に、必ずこのコマンドの実際の出力を確認する。

## Git の運用

- **ブランチ**：`feat/`、`fix/`、`docs/`、`chore/` を接頭辞にした機能単位のブランチ
- **コミット**：Conventional Commits 形式。1コミット1意図で分割する（例: マイグレーション、IMAP層、コマンド、UIの4コミット）
- **PR**：1つの変更目的に限定する。本文に変更概要、テスト結果の数値、既知の制限を書く

### Stacked PR

互いに依存する変更や、同じファイルを触る変更は、ブランチを数珠つなぎにした Stacked PR にする。
独立に作ったPR同士でも、後から衝突が見つかったらリベースして一本のスタックに再構成する。
今回は既読管理が `Mail` 構造体を変え、送信機能が `Mail` を構築していたため、7本のPRを次の一本に統合した。

```
main ← #72(既読) ← #73(添付) ← #70(SMTP) ← #71(Compose) ← #76(削除) ← #74(IDLE) ← #75(通知)
```

スタックの各PRには親PRとマージ順を明記する。

### マージ手順

`gh pr merge --delete-branch` は、そのブランチをbaseにしている子PRを自動クローズする（#73で実際に発生し、ブランチ復元と再オープンで復旧した）。
スタックは次の順で1本ずつマージする。

1. 親PRをマージする（ブランチはまだ消さない）
2. 子PRのbaseを `gh pr edit <N> --base main` で付け替える
3. 子PRの mergeable 状態を確認してからマージする
4. ブランチの削除は全部のマージが終わってからまとめて行う

マージ後は main を pull し、全テストを流して結合状態を検証する。

## 並行サブエージェント開発

独立した機能は、git worktree で隔離したサブエージェントに並行して実装させる。
今回は最大3体を並行させ、リードは統合、レビュー、衝突解決に回った。

### タスクの切り方

エージェントへの指示には次を含める。

- 必読ファイル（agent.md、該当設計書、変更対象の既存コード）の明示
- ブランチ名と分岐元（スタックに積む場合は親ブランチを指定）
- TDDとコミット粒度、Conventional Commits の遵守
- PR作成までを含めた完了条件と、最終報告に含める項目（PR URL、テスト数、迷った判断、既知の制限）

「迷った判断」を報告させるのが要点である。
エージェントは仕様の曖昧さを自分で埋めるので、その判断を報告させないと、レビューで意図とのずれを発見できない。

### 衝突の避け方

並行開発で実際に踏んだ衝突と対策を挙げる。

- **DBマイグレーション番号**：2体のエージェントが同時に `migrate_v7` を作った。並行させる場合は番号を事前に割り当てるか、統合時にリードが改番する
- **`lib.rs` の invoke_handler**：全エージェントがコマンド登録で同じ箇所を触る。指示で挿入位置を分散させる（先頭、末尾、特定コマンドの直後）
- **同一UIファイル**：MailView を触るPRが複数あるときは、一方に「このファイルは変更しない。代わりに子コンポーネント側へ統合する」と指示する
- **ビルドキャッシュ**：worktree ごとのフルビルドは数分かかる。`CARGO_TARGET_DIR` を親リポジトリの target に向けて共有する。cargo のファイルロックで直列化されるが安全
- **vitest の収集範囲**：worktree が `.claude/worktrees/` にあると、リポジトリ全体を対象にした vitest が別ツリーのテストを拾って大量失敗する。`vitest.config.ts` の `include` を `src/**` に限定して解決した（#77）

### 統合とレビュー

エージェントの成果はそのまま信用せず、リードが diff をレビューする。
特に破壊的操作（削除）とセキュリティ境界（ファイル書き込み、認証）は重点的に読む。
レビューで見つけた問題が修正を要するほどでないなら、PRコメントとして記録し、ユーザーの判断材料にする（Gmail削除の挙動はこの経路で提起し、後日ユーザー要望として実装した）。

## バグ修正の方針

症状を抑えるパッチではなく、原因に対して修正する（`agent.md` の方針）。
今回の実例では、「未分類一覧にスレッドが出ない」という報告に対して、一覧の表示だけを直すのではなく、調査で見つかった根本の2点（未分類クエリがfolderを見ていない、送信済みメールが分類対象に混入する）を同じPRで修正した。

動作確認で問題が報告されたら、修正して即マージし、ユーザーが確認を再開できる状態を最短で作る。

## ドキュメントの運用

- **README.md**：概要とクイックスタート。詳細は SETUP.md と FEATURES.md へ誘導する
- **FEATURES.md**：実装済み機能と既知の制限。記載内容は実装から検証する（推測で書かない）
- **docs/BACKLOG.md**：優先度付きの残タスク。各項目から該当設計書の制限の節へリンクする
- **メモリ（セッション引き継ぎ）**：次セッションのエージェントが最初にバックログを読むよう、索引に導線を置く

機能を実装したら、本体設計書（`2026-04-12-pigeon-design.md`）のフェーズ一覧に実装済みマークとPR番号を追記し、設計書と実装の対応を保つ。
