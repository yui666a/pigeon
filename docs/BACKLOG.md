# Pigeon バックログ（2026-07-12 時点）

2026-07-12 の大規模実装（PR #70〜#85: 送信 / 既読 / 添付 / 削除・アーカイブ / IDLE / 通知 / ドキュメント）完了時点での残タスク。各項目の背景は `docs/superpowers/specs/` の該当設計書の「制限」「将来」節に記載がある。

## 🔴 優先度: 高（構造的負債・日常利用の穴）

### 1. Sentフォルダの同期 + APPENDUID/COPYUID 対応

**現状の最大の構造的負債。** 送信メールはローカルの `uid` が `max_uid + 1` の推定値で、サーバーUIDと一致しない。このため:

- Sentメールはサーバー側の削除・アーカイブ・既読反映ができない（LocalOnly扱い）
- アーカイブ解除もサーバー反映できない（COPYUID未保存）
- 他クライアント（Gmail Web等）から送ったメールはPigeonに取り込まれず、スレッドに出ない

**対応方針**: ①IMAP APPEND/COPY の応答から UIDPLUS の APPENDUID/COPYUID を読んで正しいUIDを保存 ②Sentフォルダも同期対象に加える（`sync_account_inner` はINBOX決め打ち。message_id のUNIQUE制約で送信時ローカル保存分と自然にマージされる設計は考慮済み）。関連: `2026-07-12-mail-send-design.md` / `2026-07-12-mail-delete-archive-design.md`

### 2. 添付ファイルの送信 + リッチテキスト（TipTap）

送信はプレーンテキストのみ。設計書 `2026-07-12-mail-send-design.md` のスコープ外リストに明記。lettre はマルチパート対応済みなので、compose UIへのファイル添付とHTML化（TipTap導入）が本体。

### 3. 複数選択の一括操作

一括削除・一括アーカイブ・一括分類が未対応。メール数が増えると必須になる。

### 4. 下書き保存（Drafts同期）

compose を閉じると入力が消える。ローカル下書き→IMAP Draftsフォルダ同期の順で段階実装可。

## ⚠️ 引き継ぎ時の注意（開発運用）

- **`gh pr merge --delete-branch` はそのブランチをbaseにする子PRを自動クローズする**。スタックのマージは「親マージ → 子のbaseをmainへ変更 → 最後にブランチ削除」の順で
- `cargo fmt -- <file>` はファイル限定にならずクレート全体を整形する
- 並行エージェント開発時: DBマイグレーション番号の事前割り当て（v7衝突の実績あり）、`lib.rs` invoke_handler への挿入位置の分散、`CARGO_TARGET_DIR` 共有でビルド高速化
- vitest は `include: src/**` 限定済み（#77）だが、それ以前のブランチでは `--dir src` が必要
- DB接続は `Mutex<Connection>` 単一。大量データでロック競合が見えたら接続プール化を検討
