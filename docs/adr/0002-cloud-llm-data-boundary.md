# ADR 0002: クラウドLLMへ送信するデータの境界

## ステータス

確定（2026-07-14）

このADRは、クラウドLLMへ送信してよいデータの境界（値・ポリシー）に関する **唯一の正（Single Source of Truth）** である。境界に関わる値やポリシーを変更する場合は、まず本ADRを改訂し、各設計書・実装・UIは本ADRを参照する形にすること（詳細は「影響」を参照）。

## コンテキスト

Pigeonはメールを案件ごとに自動グルーピングするデスクトップメールクライアントであり、分類・案件ダイジェスト生成などにLLMを用いる。デフォルトのLLMプロバイダはローカルで動作するOllamaであり、この場合データはマシンの外に一切出ないため、送信データの範囲は問題にならない。

一方、ローカルLLM（`llama3.1:8b` 等）の分類精度には限界があるため、実運用ではクラウドLLM（Anthropic直のClaude API、あるいはGoogle Cloud (Vertex AI) 上のClaude）の利用が現実的な選択肢となる。クラウドLLMを選択した瞬間、メール本文・案件資料といった機微データが第三者のネットワークに送信されうる。したがって「クラウドLLMへ送信してよいデータは何か」を厳密に定義し、プライバシーとデータ最小化を担保する必要がある。

さらに、案件ディレクトリ連携機能により、ローカルPC上の案件資料（図面・香盤表・スケジュール・メモ等）から抽出したコンテキストを分類の手がかりに使えるようになった。これらはメール本文以上に機微な情報を含みうるため、送信可否をユーザーが明示的に制御できる仕組みが不可欠である。

### 分散更新の問題（本ADR作成の直接の動機）

この境界を定める「本文冒頭N文字」という値は、かつて300文字だった。分類精度改善のためこれを1000文字へ引き上げた際、次の複数箇所を手作業で同時に修正する必要が生じた。

- `agent.md`（セキュリティルールの記述）
- 案件ディレクトリ連携の設計書（本文文字数の整合）
- LLMプロバイダ選択の設計書（クラウド警告バナーの文言）
- UIの警告バナー実装

ADRという単一の権威ある拠り所が無かったため、境界の定義が複数のドキュメント・コード・UIに分散し、変更のたびに「どこを直せば漏れがないか」を人手で追跡せざるを得なかった。これは境界の値が食い違って残る（あるドキュメントは300文字、別のUIは1000文字）リスクを生む。本ADRはこの分散を解消し、境界の定義を1箇所に集約するために作成する。

## 決定

### 1. クラウドLLMへ送信してよいデータ

クラウドLLMへ送信してよいのは、次に限定する。

| データ | 送信可否 | 補足 |
|------|---------|------|
| 件名（subject） | 送信可 | |
| 送信者（from_addr、表示名付きアドレス） | 送信可 | 案件の代表送信者リストも含む |
| 本文冒頭1000文字 | 送信可 | マルチバイトは `chars()` ベースで数える（現行値: 1000文字） |
| 案件ディレクトリ連携のコンテキスト | 条件付きで送信可 | 送信可否ポリシー（後述）に従い、ユーザーが明示的に許可したもののみ |

### 2. クラウドLLMへ送信してはいけないデータ

| データ | 理由 |
|------|------|
| 本文全文（冒頭1000文字を超える部分） | データ最小化 |
| 添付ファイルの本体 | データ最小化・機微性 |
| メールアドレスの一覧（宛先・CC等の網羅） | データ最小化 |
| 案件ディレクトリ内のファイル由来データのうち、ユーザーが明示的に許可していないもの | 名前も内容もプロンプトに含めない |

案件ディレクトリのファイル由来データは、ユーザーが明示的に許可した場合に限りクラウドLLMへ送信できる。これはユーザーの意図しない機密ファイルの外部送信を防ぐための絶対条件である。

### 3. 送信可否の3層ポリシー

案件ディレクトリ連携のコンテキストについては、次の3層で送信可否を制御する。

| 層 | 対象 | デフォルト | 制御するもの |
|----|------|-----------|-------------|
| 1. ファイル/ディレクトリ単位ルール | 案件ダイジェスト生成の入力 | 不許可 | クラウドLLMで生成する際、許可されていないファイルは名前も内容もプロンプトに含めない |
| 2. 案件単位フラグ | 分類時のコンテキスト注入 | 不許可 | 案件コンテキスト（`PIGEON-CONTEXT.md` の内容）を分類プロンプトへ注入してよいか。中身はユーザーが目で確認できるファイルなので「読んで納得したものだけ許可」という運用ができる |
| 3. ローカルLLM（Ollama）選択時 | すべて | 許可 | データがマシンから出ないため無条件で利用可 |

### 4. フェイルクローズ（判定不能なら送らない）

送信可否の判定は、危険側に倒れない（フェイルクローズ）ことを絶対条件とする。次を不変条件としてテストで担保する。

1. クラウドLLMへのプロンプト構築経路で、許可ルールに明示的にマッチしないファイルは名前すら含めない。
2. 案件単位フラグが未許可の案件のコンテキストは、クラウドLLMに注入しない。
3. 判定関数は曖昧な場合（許可ルール不在・パス不一致・状態不明・不正なパス）に常に不許可（`false`）を返す。
4. ルール評価は最長 relative_path マッチを優先する。明示的な除外（`allow = false`）が親ディレクトリの許可に勝つ。同長のときは file スコープが directory スコープに勝つ。

### 5. ローカルとクラウドで境界を分けない

本文の送信文字数（1000文字）はローカル/クラウドで分けない。精度をプライバシーの形式的制約より優先するという前提合意に基づく。ローカルLLMは元々データが外部に出ないため文字数を絞る意味が薄く、クラウド利用時も分類精度を犠牲にしない。

## 理由

### なぜデータ最小化か

クラウドLLMへ送るのは分類・ダイジェスト生成という目的の達成に必要な最小限のデータに絞る。本文全文や添付ファイル本体は分類の判定材料として費用対効果が低い一方、機微性が高い。冒頭1000文字と件名・送信者・許可済みコンテキストがあれば案件判定には十分であり、それ以上を送る必然性がない。

### なぜフェイルクローズを選ぶか

プライバシー事故は不可逆である。一度クラウドへ送信されたデータは取り消せない。したがって「判定に迷ったら送る」（フェイルオープン）ではなく「判定に迷ったら送らない」（フェイルクローズ）を採る。判定ロジックのバグ・ルールの記述漏れ・想定外のパスといった不確実性は、すべて「送らない」側に倒すことで、最悪でも「精度が上がらない」で済み、「機密が漏れる」には至らないようにする。

案件ディレクトリの許可を「ファイル行への焼き込み」ではなく「ルール方式」で表現しているのも同じ理由による。ディレクトリ許可後に追加されたファイルへ許可が誤って伝播する事故を避け、ユーザーの意図（ルール）を正本として実ファイルを毎回評価する。

## 却下した代替案

### 本文全文の送信

案件判定の精度は理論上わずかに上がりうるが、機微データの送信量が青天井になり、データ最小化の原則に反する。冒頭1000文字で分類には十分という判断から却下。

### フェイルオープン（判定不能なら送る）

実装は単純になるが、プライバシー事故が不可逆であることと相性が悪い。判定ロジックのバグやルールの記述漏れが即座に機密漏洩につながるため却下。

### ローカルとクラウドで文字数を分ける（クラウドは短く）

プライバシー配慮としては筋が通るが、分類精度がクラウド利用時に劣化する。精度をプライバシーの形式的制約より優先する前提合意に反し、また「境界の値が2つになる」ことで分散更新問題を悪化させるため却下。

### 案件ディレクトリのコンテキストをデフォルト許可

ユーザーの利便性は上がるが、意図しない機密ファイルの外部送信という最悪の事故を招く。デフォルト不許可・明示許可のみを採用し却下。

## 影響

### 境界を変更するときの手順

今後、送信データの境界（本文文字数・送信可否ポリシー・不変条件）を変更する場合は、**まず本ADRを改訂すること**。各設計書・実装・UIは本ADRの決定を参照・反映する従属物と位置づける。これにより、境界の定義が複数箇所に分散して食い違う（かつての300→1000文字変更で起きた分散更新）状態を防ぐ。

具体的には、本ADRを正としたうえで、次の従属箇所を本ADRに合わせて更新する。

### 関連する実装の位置

- 送信可否の判定ロジック: `src-tauri/src/project_context/cloud_policy.rs`（`is_cloud_allowed`。フェイルクローズ・最長マッチ・file スコープ優先・`..` セグメント拒否を実装。不変条件はここのユニットテストで担保）
- 許可ルールの永続化: `src-tauri/src/db/cloud_rules.rs`（`project_cloud_rules` テーブルのCRUD）
- 本文文字数の定数: `src-tauri/src/models/classifier.rs`（`BODY_PREVIEW_CHARS = 1000`、`MailSummary::from_mail`）
- 分類プロンプト構築: `src-tauri/src/classifier/prompt.rs`（許可された案件コンテキストのみ注入）

### 関連するUIの位置

- クラウド警告バナー: `src/components/sidebar/LlmSettingsDialog.tsx`（クラウドLLM選択時に送信データ範囲を明示。文言中の本文文字数は本ADRの値と一致させる）
- クラウド送信設定ダイアログ: ファイル/ディレクトリ単位の許可ルール（層1）と案件単位トグル（層2）をユーザーが操作する画面

### 関連する規約

- `agent.md` のセキュリティルール（LLMへ送信するデータの記述）は本ADRの決定を反映したものであり、値を変える際は本ADRと同時に整合させる。
