# Architecture Decision Records (ADR)

このディレクトリは、Pigeon の**横断的・恒久的なアーキテクチャ意思決定**を1件1ファイルで記録する。

複数の機能・設計書にまたがる決定（例: 「クラウド LLM に何を送ってよいか」）は、放置すると各設計書・実装・UI に定義が分散し、変更のたびに手作業で同期する羽目になる。ADR はその決定の**唯一の正（Single Source of Truth）**を1箇所に定め、各設計書・実装は ADR を参照する従属物として扱う。

機能単位の詳細設計は `docs/design/`、実装計画は `docs/plans/`、実装完了で役目を終えたものは `docs/archive/` にある。ディレクトリの使い分けとライフサイクルは [`docs/DEVELOPMENT-STYLE.md`](../DEVELOPMENT-STYLE.md) を参照。

## 運用

### いつ ADR を書くか

- ある値・ポリシー・方針が**複数のファイルに散りそう**なとき、または既に散っているとき
- 「なぜ別のやり方を採らなかったか（却下案）」がコードからは復元できないとき
- フェーズや機能を横断して効く決定（セキュリティ境界、抽象化方針、同期規約など）

単一機能に閉じた設計判断は ADR ではなく `docs/design/` の設計書に書けばよい。

### 書き方

- ファイル名は連番 + ケバブケース（`NNNN-短い題名.md`）。番号は既存の最大値 + 1。
- 各 ADR は次のセクションを持つ: **ステータス / コンテキスト / 決定 / 理由 / 却下した代替案 / 影響 / 参照**。
- 決定を変えるときは、当該 ADR を**改訂**する（新しい番号で置き換える場合は旧 ADR のステータスを「置換済み（→ NNNN）」にし、履歴を残す）。
- 参照は移動耐性のため、パスではなく設計書の**名称**で書くことを推奨する（`docs/` 配下は再編されうる）。

### ステータスの語彙

- **確定** — 決定済みで実装に反映されている（または反映中）
- **進行中** — 決定の方向は合意済みだが、実装が段階的に進行中
- **置換済み（→ NNNN）** — 後続の ADR に置き換えられた。歴史的経緯として残す
