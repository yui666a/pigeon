# ADR 0006: 起動パスのブロッキング禁止とデータ取得量の上限

## ステータス

確定（2026-07-20）

## コンテキスト

実機で次の3つの体感遅延が報告された。いずれもユーザーが操作を待たされる、または UI が完全に無反応になる症状である。

1. アプリ起動後、ウィンドウが表示されるまで数十秒かかる
2. アカウント選択後、案件一覧とスレッド一覧の表示が遅い
3. 設定ボタンを押すと数秒 UI がフリーズする

コードを調査したところ、原因は個別の実装ミスではなく、**「重い処理をいつ・どのスレッドで実行するか」「一度に何件取得するか」という2つの規範が存在しなかったこと**に起因していた。

### 起動パスに同期ブロッキングが積まれていた

`run()` の冒頭、`tauri::Builder` を構築する**前**に、以下が同期実行されていた。

- OS キーチェーンからのマスターキー読み出し
- `SecureStore::open_with_migration()` による Stronghold スナップショットの復号

後者が支配的で、コスト源は `iota_stronghold::KeyProvider` の鍵導出（意図的に重い KDF）である。**保管している秘密の件数に依存せず固定で数十秒**かかる。さらに鍵移行が必要な場合は旧鍵での再試行が直列に走り、所要時間が倍増する。

これらが Builder 構築前にあるため、Tauri のイベントループもウィンドウも存在しない状態で待つことになる。ユーザーからは「アイコンをクリックしたが何も起きない」と見え、スプラッシュ画面を表示する余地すらない。

そして調査で判明した決定的な事実として、**`SecureStore` は起動時には一切使われていない**。`manage` で登録されるだけで、実際の利用は `sync_account` の資格情報解決などユーザー操作時である。起動パスに置く必然性が最初からなかった。

### 同期コマンドが UI スレッドを直接ブロックしていた

Tauri v2 では `async fn` でない `#[tauri::command]` は**メインスレッド（UI イベントループ）上で同期実行される**。`get_llm_settings` / `set_llm_settings` が同期関数のまま Stronghold を参照していたため、Stronghold のロック待ちがそのまま WebView のフリーズとして現れていた。

さらに `get_llm_settings` は DB ロックを保持したまま Stronghold のロックを待つ入れ子構造になっていた。`DbState` のドキュメントコメントは「クロージャの間ロックを保持するため、await を挟む処理には使わないこと」と警告しており、この規約が守られていなかった。

Stronghold 側のロック競合が長引く根本原因は、`insert` / `delete` のたびに `commit_with_keyprovider` でスナップショット全体を再暗号化して書き出す実装にある。1回の書き込みコストが数十秒級であり、その間 `get` は待たされる。

### 取得件数に上限がなかった

スレッド一覧・未分類一覧の取得クエリが `LIMIT` を持たず、INBOX 全件を `body_text` / `body_html` 込みで取得していた。実機のメール件数は 10,000 件規模であり、本文全体が IPC で JSON シリアライズされてフロントへ渡っていた。

フロント側には表示件数を絞る仕組み（`useDisplayLimit`）が存在するが、これは「データは全件保持し、描画のみ先頭 N 件で切る」方式である。DB 取得・IPC 転送・JSON パースのコストは一切削減されていない。

### キャッシュがあるのにネットワークを待っていた

スレッド一覧の初回取得が、IMAP 同期の完了を待ってからローカル DB の読み出しを行う逐次チェーンになっていた。ローカル DB に既にメールが存在するにもかかわらず、ネットワーク往復を待ってから描画していた。

## 決定

### 1. 起動パスに同期ブロッキング処理を置かない

`tauri::Builder` 構築前に実行してよいのは、**失敗したらアプリが起動できない、かつ短時間で完了する処理のみ**とする。具体的にはデータディレクトリ作成、DB 接続、マイグレーションに限る。

秘密情報ストアの初期化のように、**起動直後には不要で、かつ所要時間が長い処理は遅延初期化する**。`SecureStore` は `OnceCell` 等で包み、最初に秘密情報を必要とする操作の時点で初期化する。

判断基準は「その処理の結果が、ウィンドウを表示するために必要か」である。必要でないなら起動パスに置かない。

### 2. 重い処理を行うコマンドは async にする

`#[tauri::command]` が以下のいずれかに該当する場合、`async fn` として定義し、ブロッキング処理は `spawn_blocking` に委ねる。

- SecureStore（Stronghold）にアクセスする
- ネットワーク I/O を行う
- OS キーチェーンにアクセスする
- 完了時間が数十 ms を超えうる DB クエリを実行する

同期コマンドは UI スレッドを直接ブロックするため、上記に該当する処理を同期コマンドに置くことを禁止する。

### 3. DB ロックを保持したまま他のロックを待たない

`with_conn` のクロージャ内から SecureStore にアクセスしてはならない。秘密情報の参照が必要な場合は、DB ロックを取得する**前**に評価を済ませる。

ロック取得順序の入れ子は、待ち時間の合成とデッドロックの温床になる。

### 4. 秘密情報の書き込みでスナップショット全体を再暗号化しない

`insert` / `delete` のたびに全体を再暗号化してコミットする実装を改め、書き込みコストを削減する。この書き込みコストは `get` 側のロック待ち時間として波及するため、読み取り性能の前提条件でもある。

### 5. 一覧取得クエリは必ず上限を持つ

メール・スレッドの一覧を返すクエリは `LIMIT` を持たなければならない。`LIMIT` なしの全件取得は、件数がユーザーのメール蓄積量に比例して増加するため、時間の経過とともに必ず破綻する。

表示件数の制御はフロントの描画段階ではなくサーバ（Rust）側の取得段階で行う。フロントでの `slice` は転送コストを削減しないため、性能対策として扱わない。

### 6. ローカルキャッシュの描画をネットワーク同期の完了に依存させない

ローカル DB に取得済みデータが存在する場合、それを先に描画する。ネットワーク同期はその後に実行し、完了後に差分を反映する。

キャッシュがあるのにネットワークを待つ設計は、オフライン時と低速回線時に体感を著しく損なう。

## 理由

いずれの決定も「症状に対する個別対処」ではなく「同じ事故の再発を構造的に防ぐ規範」として設定している。

起動パスの規範（決定 1）は、今後 SecureStore 以外の重い初期化処理が追加されたときにも同じ判断を適用できる。実際、Ollama への疎通確認や IMAP 接続は既に `setup()` 内で `spawn` されており正しく実装されていた。規範を明文化することで、この正しい実装が偶然ではなく規約になる。

async 化の規範（決定 2）は、Tauri v2 の実行モデルに起因する落とし穴である。同期コマンドがメインスレッドで走る事実はドキュメントに書かれているが、コードを読んだだけでは気づきにくい。ADR に残すことで、新しいコマンドを追加する際のチェック項目になる。

取得上限の規範（決定 5）は、現時点で 10,000 件という具体的な規模が判明したことが根拠である。フロントの `useDisplayLimit` は描画最適化としては正しく機能しているが、これを性能対策と誤認すると転送コストの問題が見過ごされる。両者の役割を明確に分離する。

## 却下した代替案

### 起動時にスプラッシュ画面を出す

Stronghold の初期化を起動パスに残したまま、待ち時間中にスプラッシュを表示する案。

却下する。ウィンドウ生成自体が Builder 構築後であるため、Builder 前のブロッキングにはスプラッシュを出す余地がない。仮に構造を変えて表示できたとしても、数十秒待たされる事実は変わらず、症状を隠蔽するだけである。

### 同期コマンドのまま実行を高速化する

Stronghold の KDF パラメータを弱めて初期化を高速化する案。

却下する。KDF の強度は暗号化ストアの安全性そのものであり、性能のために下げるのは ADR 0003 が定めた秘密情報保護の水準を損なう。性能問題は実行タイミングの設計で解決すべきであり、暗号強度とトレードオフしてはならない。

### フロント側だけでローディング表示を改善する

各画面にスピナーを出し、体感を改善する案。

却下する。UI がフリーズしている原因は Rust のメインスレッドブロックであり、JavaScript は正常に動作していない。スピナーを追加しても描画されない。加えて、待ち時間そのものは短縮されない。

### 取得上限をフロントの `useDisplayLimit` に任せる

現状維持案。

却下する。前述のとおり `useDisplayLimit` は全件をメモリに保持した上で描画のみを制限する実装であり、DB 取得・IPC 転送・JSON パースのコストを削減しない。メール件数の増加に対して線形に悪化する。

## 影響

### コードへの影響

- `SecureStore` の初期化が遅延化されるため、初回の秘密情報アクセス時に待ち時間が移動する。決定 4 の実施によりこの待ち時間自体も短縮される
- 秘密情報に触れるコマンドの多くが `async fn` に変更される。呼び出し側（フロント）は既に `invoke` を await しているため、フロントの変更は不要
- 一覧取得コマンドのシグネチャに件数上限が追加される。フロントは追加取得（ページング）の考慮が必要になる

### スキーマへの影響

未読件数の集計クエリが `mails` テーブルの全行スキャンになっている問題があり、`(account_id, folder, is_read)` に対するインデックスを追加する。マイグレーション v23 を消費する。

### 運用への影響

新しい `#[tauri::command]` を追加する際、決定 2 のチェックリストに照らして同期・非同期を判断する。新しい一覧取得クエリを追加する際、決定 5 に従い `LIMIT` を必須とする。

### 本 ADR の射程外

案件一覧取得における案件ごとの IPC 往復（N+1）は、案件数が 15 件規模の現状では体感差を生まない。取得方法の集約は行うが、これは決定 5 の適用ではなく個別の実装改善として扱う。案件数が数百規模に育った場合は本 ADR の決定 5 の対象として再検討する。
